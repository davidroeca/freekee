# KDBX Compatibility Matrix

What `freekee` does and does not preserve when round-tripping a KeePassXC-generated KDBX file. Updated per release.

The source of truth is `tests/roundtrip/fixtures/`. Every entry below maps to one fixture directory there.

Read = the fixture parses without error.
Round-trip = `read -> write -> read` is structurally equivalent (see `crates/kdbx/tests/common/mod.rs::assert_self_roundtrip`).
Verify = a file we wrote can be re-opened by `keepassxc-cli` (gated on the `keepassxc-verify` feature). Init = `freekee init`-created file (no fixture) passes `keepassxc-cli db-info`.

## Known upstream gaps in `keepass = 0.12.0`

1. **`PreviousParentGroup` (KDBX 4.1) - read failure.** See `groups-and-entries` row above. Affects any database where a group or entry has been moved. Upstream tracking: [issue #303](https://github.com/sseemayer/keepass-rs/issues/303); maintainer fix in flight at [PR #308](https://github.com/sseemayer/keepass-rs/pull/308). Once that ships and we bump `keepass`, un-ignore `roundtrip_groups_and_entries_preserves_hierarchy`.

2. **Writer produces files KeePassXC rejects - partially worked around.** `keepass-rs` 0.12.1 ships the `cs_opt_bool` ("null") and `cs_opt_intbool` (0/1) fixes from [PR #312](https://github.com/sseemayer/keepass-rs/pull/312), so KeePassXC now accepts files that were read from its own fixtures and re-saved through our code. However, `cs_opt_fromstr::serialize(None)` still writes empty elements (`<IconID></IconID>`, `<UsageCount></UsageCount>`) that KeePassXC rejects with "Invalid number value". We work around this in `kdbx::Database::save` via `pre_save_normalize()`, which fills in `Some(...)` defaults for any `None` numeric fields before calling the upstream serializer. The correct upstream fix is `skip_serializing_if = "Option::is_none"` on every `cs_opt_fromstr` field, proposed at [issue #311](https://github.com/sseemayer/keepass-rs/issues/311).
   1. `cs_opt_bool::serialize(None)` - **fixed upstream in 0.12.1** (emits `"null"` instead of empty element).
   2. `AutoType::data_transfer_obfuscation` - **fixed upstream in 0.12.1** (uses `cs_opt_intbool`, emits `"0"`/`"1"`).
   3. (Cosmetic, warnings-only) Casing drift on `<DefaultUserName>`, `<DefaultUserNameChanged>`, `<ProtectUserName>` - keepass-rs writes them as `Username` (lowercase n). KeePassXC ignores them as unknown elements. Not yet fixed upstream.
   4. `cs_opt_fromstr` numeric fields (`IconID`, `UsageCount`, `MaintenanceHistoryDays`, etc.) - **not fixed upstream**; `None` still serializes as `<Tag></Tag>`. Worked around in our `pre_save_normalize()`.

3. **KDBX minor version normalized to 0 on parse.** `Database::config.version` always reports `KDB4(0)` after parsing, regardless of whether the file header says `KDB4(0)` or `KDB4(1)`. Visible via `cargo run --features dump-expected --bin dump-expected` against the `empty` fixture (header is KDBX 4.1, dump reports `kdb4.0`). Cosmetic for round-trip - both directions read consistently - but means our own writes are always emitted as KDBX 4.0. Suitable for a follow-up upstream issue (not yet filed).

## Mutation-path semantics (M1)

Every entry mutation routes through upstream change-tracking machinery so the resulting file behaves correctly under KeePassXC sync:

- **Field edits use `EntryMut::edit_tracking`.** The wrapper's `Database::set_entry_field` and the `set` / `rotate entry` rotations all funnel through this closure, which snapshots the prior `Entry` into `History` and stamps `times.last_modification` automatically. Bypassing it (writing `entry.fields[...]` directly) would silently drop the audit trail KeePassXC users expect.
- **Deletes use `EntryTrack::remove`, not the plain `EntryMut::remove`.** Only the tracking variant adds the entry's UUID to the database's `deleted_objects` registry. Without it KeePassXC treats the missing entry as "newer state on the other side" and resurrects it on the next merge. The wrapper's `Database::remove_entry` hides this trap; `Database::deleted_object_count()` lets tests confirm the tombstone landed.
- **Moves and renames also flow through `track_changes`** so cross-group relocation and same-group rename both record a history snapshot and an updated `last_modification`.
- **`Database::save` regenerates the KDF salt and IVs from `getrandom` on every call.** That is the only "rekey" primitive freekee uses - `rotate passphrase` is just a `save` with the new passphrase, and `rotate kdf-params` mutates `config.kdf_config` and re-saves so the next derivation uses the new params plus a fresh salt.

**Keyfile composition (M2 phase 1, resolved).** `kdbx::Database::save` now takes an optional keyfile path and writes the same composite the file was opened with. `core::Vault` holds the keyfile path across the vault's lifetime and threads it into both save and verify, so passphrase / KDF / entry rotations preserve the keyfile binding. New: `core::Vault::rotate_keyfile(Option<&Path>, RotateOpts)` adds, replaces, or removes the keyfile composite, surfaced as `freekee rotate keyfile [--new-keyfile <PATH> | --remove]`. The earlier `freekee init --keyfile` rejection is gone; init wires the keyfile through to the new file.
